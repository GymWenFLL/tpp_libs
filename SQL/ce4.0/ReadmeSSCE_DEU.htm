1.0 Einführung

Microsoft® SQL Server™ Compact 4.0 SP1 ist ein In-Process-Datenbankmodul mit geringem Ressourcenbedarf, mit dem Entwickler stabile Anwendungen für Windows-Desktopcomputer und für Einsteiger-ASP.NET-Websites erstellen.

1.1 Zugreifen auf SQL Server Compact 4.0 SP1

Die Servertools von SQL Server Compact 4.0 SP1 können von folgendem Speicherort im Web heruntergeladen werden:

3.0 Bekannte Probleme

3.1

Die systemeigenen Binärdateien (DLL) von SQL Server Compact 4.0 SP1 benötigen die Microsoft Visual C++ 2008-Laufzeitbibliotheken (x86 und x64), Service Pack 1, um ordnungsgemäß ausgeführt werden zu können. Bei der Installation von SQL Server Compact 4.0 SP1 mit der Installationsdatei werden auch die Visual C++ 2008 SP1-Laufzeitbibliotheken installiert. Wenn SQL Server Compact 4.0 SP1 im Anwendungsordner privat bereitgestellt wird, vergewissern Sie sich, dass die Ordner x86 und amd64, die sich unter %Program Files%\Microsoft SQL Server Compact Edition\v4.0\Private befinden, für die Bereitstellung der systemeigenen Binärdateien von SQL Server Compact verwendet werden, da die Ordner x86 und amd64 die erforderlichen systemeigenen Binärdateien für SQL Server Compact und für die Microsoft Visual C++ 2008-Laufzeitbibliotheken (x86 und x64), Service Pack 1, enthalten. Weitere Informationen finden Sie in der SQL Server Compact 4.0-Onlinedokumentation. Die Microsoft Visual C++ 2008-Laufzeitbibliotheken (x86 und x64), Service Pack 1, werden auch von den folgenden Anwendungen installiert: 

  1. Bei der Installation von .NET Framework 3.5 SP1 werden auch die Visual C++ 2008 SP1-Laufzeitbibliotheken installiert.

  2. Visual C++ 2008 SP1-Laufzeitbibliotheken können unter der folgenden URL heruntergeladen und installiert werden:
    Visual C++ 2008

Wenn .NET Framework 2.0, .NET Framework 3.0, .NET Framework 4 oder .NET Framework 4.5 Beta installiert wird, werden die Visual C++ 2008-Laufzeitbibliotheken (x86 und x64), Service Pack 1, nicht installiert.

3.2

Wenn das Installationsprogramm für die SQL Server Compact-Laufzeit 4.0 SP1 auf einem Computer ausgeführt wird, auf dem keine Version von .NET Framework installiert ist, wird die Installation ohne Fehler fortgesetzt, wobei jedoch einige Schritte nicht ausgeführt werden. Dies kann auf einem entsprechenden Computer zu Fehlern führen. Folgende Schritte werden nicht ausgeführt:

  • Der machine.config-Eintrag für .NET erhält keinen Eintrag für SQL Server Compact als Datenanbieter.

  • Die relevanten Assemblys der SQL Server Compact-Laufzeit werden nicht im globalen Assemblycache platziert.

Dies kann auch der Fall sein, wenn ein Benutzer die folgenden Schritte ausführt:

  • Installieren der SQL Server Compact 4.0 SP1-Laufzeit

  • Installieren von .NET Framework 3.5 SP1, 4 oder 4.5 Beta

Um das Problem zu beheben, installieren Sie die SQL Server Compact-Laufzeit nach der Installation von .NET Framework neu, oder reparieren Sie sie.

3.3

SQL Server Compact 4.0 SP1 erkennt die Encrypt-Eigenschaft der SqlCeConnection-Klasse nicht, und die Eigenschaft sollte nicht zur Verschlüsselung von Datenbankdateien verwendet werden.

Hinweis

Die Verschlüsseln-Eigenschaft wurde in der Version SQL Server Compact v3.5 als veraltet markiert. Diese Eigenschaft wurde in SQL Server Compact 4.0 SP1 nur aus Gründen der Abwärtskompatibilität beibehalten.

Verwenden Sie die Verschlüsselungsmodus-Eigenschaft der SqlCeConnection-Klasse, um SQL Server Compact 4.0 SP1-Datenbankdateien zu verschlüsseln. In den folgenden Beispielen wird die Verwendung der Verschlüsselungsmodus-Eigenschaft veranschaulicht.

  • So erstellen Sie eine neue verschlüsselte SQL Server Compact 4.0 SP1-Datenbank

    C#:

     
    SqlCeEngine engine = new SqlCeEngine("Data Source=Northwind.sdf;encryption mode=platform default;Password=<enterStrongPasswordHere>;");
engine.CreateDatabase();

    Visual Basic:

     
    Dim engine As SqlCeEngine = New SqlCeEngine("Data Source=Northwind.sdf;encryption mode=platform default;Password=<enterStrongPasswordHere>;")
engine.CreateDatabase()

  • So ändern Sie den Verschlüsselungsmodus einer vorhandenen SQL Server Compact 4.0 SP1-Datenbank

    C#:

     
    SqlCeEngine engine = new SqlCeEngine("Data Source=Northwind.sdf;Password=<enterStrongPasswordHere>;");
engine.Compact("Data Source=Northwind.sdf;encryption mode= enginedefault;Password=<enterStrongPasswordHere>;");

    Visual Basic:

     
    Dim engine As SqlCeEngine = New SqlCeEngine("Data Source=Northwind.sdf;Password=<enterStrongPasswordHere>;") 
engine.Compact("Data Source=Northwind.sdf;encryption mode= engine default;Password=<enterStrongPasswordHere>;")

  • So verschlüsseln Sie eine unverschlüsselte SQL Server Compact 4.0 SP1-Datenbank

    C#:

     
    SqlCeEngine engine = new SqlCeEngine("Data Source=Northwind.sdf");
engine.Compact("Data Source=Northwind.sdf;encryption mode=platform default;Password=<enterStrongPasswordHere>;");

    Visual Basic:

     
    Dim engine As SqlCeEngine = New SqlCeEngine("Data Source=Northwind.sdf;") 
engine.Compact("Data Source=Northwind.sdf;encryption mode=platform default;Password=<enterStrongPasswordHere>;"

3.4

SQL Server Compact 4.0 SP1-Datentypen für Parameter wie SqlDbType oder DbType sollten ausdrücklich festgelegt werden.

Wenn die Datentypen für Parameter wie SqlDbType oder DbType nicht ausdrücklich festgelegt werden, wird eine Ausnahme ausgegeben. Um dieses Problem zu umgehen, müssen Sie den Datentyp für Parameter wie SqlDbType oder DbType ausdrücklich festlegen. Im Fall von BLOB-Datentypen (image und ntext) ist dies kritisch. Ein Codebeispiel finden Sie unten:

C#:

 
SqlCeEngine engine = new SqlCeEngine(connString);
engine.CreateDatabase();
engine.Dispose();

SqlCeConnection conn = new SqlCeConnection(connString);
conn.Open();

SqlCeCommand cmd = conn.CreateCommand();
cmd.CommandText = "CREATE TABLE BlobTable(name nvarchar(128), blob ntext);";
cmd.ExecuteNonQuery();

cmd.CommandText = "INSERT INTO BlobTable(name, blob) VALUES (@name, @blob);";
SqlCeParameter paramName = cmd.Parameters.Add("name", SqlDbType.NVarChar, 128);
SqlCeParameter paramBlob = cmd.Parameters.Add("blob", SqlDbType.NText);
paramName.Value = "Name1";
paramBlob.Value = "Name1".PadLeft(4001);

cmd.ExecuteNonQuery();

Visual Basic:

 
Dim engine As SqlCeEngine = New SqlCeEngine(connString)
engine.CreateDatabase()
engine.Dispose()

Dim conn As SqlCeConnection = New SqlCeConnection(connString)
conn.Open()

Dim cmd As SqlCeCommand = conn.CreateCommand()
cmd.CommandText = "CREATE TABLE BlobTable(name nvarchar(128), blob ntext);"
cmd.ExecuteNonQuery() 

cmd.CommandText = "INSERT INTO BlobTable(name, blob) VALUES (@name, @blob);"
Dim paramName As SqlCeParameter
Dim paramBlob As SqlCeParameter
paramName = cmd.Parameters.Add("name", SqlDbType.NVarChar, 128)
paramBlob = cmd.Parameters.Add("blob", SqlDbType.NText)
paramName.Value = "Name1"
paramBlob.Value = "Name1".PadLeft(4001)

cmd.ExecuteNonQuery()

3.5

Die Version SQL Server Compact 4.0 SP1 unterstützt keine Synchronisierung von Daten mit SQL Server mithilfe von Technologien wie Microsoft Sync Framework oder Mergereplikation bzw. Remotedatenzugriff (Remote Data Access, RDA).

3.6

Die Generierung des Entitätsdatenmodells für SQL Server Compact SP1 schlägt fehl, wenn das SQL Server Compact SP1-Schema doppelte Einschränkungsnamen enthält.

In SQL Server Compact sind die Einschränkungsnamen in einer Tabelle eindeutig, und daher sind doppelte Einschränkungsnamen in der Datenbank zulässig. Das Verhalten unterscheidet sich von SQL Server, wo die Einschränkungsnamen in der gesamten Datenbank eindeutig sind. Wenn ein SQL Server Compact-Schema Einschränkungsnamen mit doppelter referenzieller Integrität (Primärschlüssel – Fremdschlüsselbeziehung) beinhaltet, schlägt die Generierung des Entitätsdatenmodells mit dem Entity Data Model-Assistenten im ADO.NET Entity Framework fehl. Dieses Problem kann umgangen werden, indem für die Datenbank ein eindeutiger Name der doppelten Einschränkung festgelegt wird, z. B. durch Hinzufügen des Tabellennamens zum Einschränkungsnamen.
